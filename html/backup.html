Backup

3-2-1 원칙

  • 3벌 이상의 사본 (원본 + 로컬 서버 + 클라우드)
  • 2가지 다른 매체 (로컬 ZFS + 클라우드 오브젝트 스토리지)
  • 1벌은 오프사이트 (화재·도난·서버 전체 손실에도 생존)

다층 구조

같은 데이터를 성격이 다른 여러 도구로 백업해 단일 실패점을 없앤다.

계층 도구 성격 대상
1 rsync 단순 미러 (그대로 복사) 로컬 서버
2 restic 중복제거 + 암호화 스냅샷 로컬 서버 + 오프사이트
3 kopia 두 번째 중복제거 엔진 로컬 서버
4 클라우드 S3 호환 오브젝트 스토리지 오프사이트

작업 파일(work/downloads)은 rsync + restic(로컬) + restic(오프사이트) + kopia = 4중 백업.

자동화 원칙

  • 스케줄: launchd(macOS)/cron(Linux)로 새벽 시간대 자동 실행
  • 알림: 매 실행 결과를 메신저로 통보 (성공/실패/스킵), 아침에 전체 요약 1건
  • 외장 드라이브: 미연결 시 실패가 아니라 스킵 처리 (이동식 매체 특성)
  • 검증: 주 1회 --checksum으로 양쪽 재해시해 silent corruption 감지

실전에서 부딪힌 것들

파일명이 백업을 깨뜨린다

  • 길이 초과: 초장문 제목 → Linux 255바이트 한계 초과 → rsync 실패. 다운로드 시점에 제목 길이를 제한(yt-dlp -o "%(title).150B...")해 원천 차단.
  • 이모지: 파일명의 이모지 → rsync의 iconv가 변환 실패 → exit 23. BMP 밖 문자를 다운로드 시 제거.
  • NFD/NFC: macOS는 한글을 자모 분해(NFD)로 저장, Linux는 완성형(NFC). rsync --iconv=utf-8-mac,utf-8 + 기존 파일은 convmv로 일괄 정규화.

속도 튜닝은 병목 분리부터

백업이 느릴 때 구간별 최대치를 각각 측정해 진짜 원인만 고쳤다.

소스(USB) → Mac → [네트워크] → [ssh 암호화] → [서버 디스크]
