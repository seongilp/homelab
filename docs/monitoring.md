# Monitoring

## 알림 우선, 대시보드는 보조

홈랩 모니터링은 두 축으로 나눈다.

- **알림(Push)** — 문제가 생겼을 때 **먼저 나를 찾아오는** 것. 24/7 지켜볼 수 없으니 이게 핵심.
- **대시보드(Pull)** — 궁금할 때 **내가 찾아가서 보는** 것. 상태 파악·튜닝 근거용.

## 대시보드: Beszel

경량 셀프호스팅 모니터링(단일 Go 바이너리 허브 + 초경량 에이전트). 5노드를 한 화면에서.

```
허브: ebs (상시 가동 노드)        ← 감시자는 감시 대상 밖에 둔다
 ├─ ebs      (에이전트)           원칙: prodesk가 죽을 때
 ├─ prodesk  (에이전트)           죽은 prodesk가 알림을 보낼 순 없다
 ├─ msg10p   (에이전트)
 ├─ freebsd  (에이전트, NAT 뒤)
 └─ fb-crnt  (에이전트, NAT 뒤)
```

**왜 ebs가 허브인가** — 모니터링 허브야말로 24/7 떠 있어야 하는 대표적 상시 서비스이고,
감시자(허브)는 감시 대상과 물리적으로 분리되어야 한다. 이미 "상시 서비스" 담당인
저전력 박스(ebs)가 적임. Beszel 허브는 초경량이라 12년 된 i3로도 충분.

**왜 Netdata/Prometheus가 아닌가** — Netdata는 노드당 무겁고(약한 ebs엔 부담),
Prometheus+Grafana는 셋업·관리가 무겁다. 5노드 홈랩엔 Beszel의 "단일 바이너리 + 경량"이 맞다.

### NAT 뒤 VM 연결 (트러블슈팅 기록)

FreeBSD 학습 VM 2대는 하이퍼바이저의 libvirt NAT(`192.168.122.x`) 안에 있어
허브(다른 서브넷)에서 직접 못 닿는다. 하이퍼바이저에서 포트 포워딩으로 노출.

```bash
# DNAT만으론 부족 — 응답이 VM→허브로 직접 가서 asymmetric routing으로 깨진다
iptables -t nat -A PREROUTING -p tcp --dport 45877 -j DNAT --to 192.168.122.201:45876
# MASQUERADE로 VM이 응답을 하이퍼바이저 경유로 되돌리게 해야 완결된다
iptables -t nat -A POSTROUTING -o virbr0 -d 192.168.122.201 -p tcp --dport 45876 -j MASQUERADE
```

교훈: NAT 뒤 서비스를 외부에 노출할 때 **DNAT(도착) + MASQUERADE(응답)**가 세트다.
DNAT만 있으면 TCP 핸드셰이크는 되는 것처럼 보여도 실제 세션이 안 선다.

## 알림: 텔레그램 (단일 채널로 통합)

백업·모니터링·서버 작업 알림을 **하나의 텔레그램 봇**으로 통합. 채널이 흩어지면 놓친다.

- **백업** — 매 실행 성공/실패/스킵 + 매일 아침 전체 요약 1건
- **모니터링(Beszel)** — 노드 다운(3분 지속) + 디스크 임계(85%) 시 push
- **서버 장기 작업** — OS 업그레이드 등 단계 전환마다 진행 알림

Beszel은 shoutrrr 형식(`telegram://<token>@telegram?chats=<id>`)으로 기존 봇에 연동.

## 그 외 상시 감시

- **ZFS 스크럽** — 매월 1일, 풀 전체를 체크섬 재검증 (bit rot 감지·복구)
- **smartd** — 디스크 SMART 속성 상시 감시 (재할당 섹터 등 사전 경고)
- **백업 검증** — 주 1회 `--checksum`으로 원본/사본 재해시
